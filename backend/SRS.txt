1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for the Personal Finance Tracker (PFT), a responsive, multi-lingual web application that helps individuals manage personal finances through income and expense tracking, budget planning, and financial analytics.
This document is the authoritative reference for all project stakeholders — business analysts, software architects, developers, QA engineers, project managers, and support teams — throughout the entire software development lifecycle. It provides sufficient detail for a development team to begin implementation without ambiguity.
1.2 Scope
The Personal Finance Tracker is a standalone, browser-based web application with no direct banking API integration in version 1.0. It provides:
    • User registration, authentication, and role-based access control
    • Multi-account financial management (cash, bank, credit card, digital wallet, savings, investment)
    • Full income and expense transaction tracking with categorization
    • Monthly budget planning with automated threshold alerts
    • Recurring transaction management with configurable frequencies
    • Financial analytics dashboard with charts and summaries
    • Monthly and yearly report generation with PDF export
    • In-app and email notification system
    • Administrative platform management interface
    • Full Arabic (RTL) and English (LTR) localization

The system is intended for individuals seeking better financial management and spending control. It is explicitly NOT in scope for version 1.0 to provide bank integrations, investment tracking, multi-currency conversion within an account, native mobile applications, or AI-driven financial advice.

2. Overall Description
2.1 Product Perspective
The PFT is a standalone, browser-based web application. In version 1.0 it operates independently from any third-party banking APIs, payment gateways, or external financial data services. All data is self-reported by users. The system consists of three primary components:
    • React-based Single-Page Application (SPA) frontend
    • Node.js / Express RESTful API backend
    • PostgreSQL relational database

External services are limited to an SMTP email provider, an optional SMS gateway for OTP, a PDF generation engine, and object storage for receipt images. The product is designed to evolve in future versions to support open-banking APIs, native mobile apps, and AI-powered financial insights.

2.3 User Classes and Characteristics
2.3.1 Guest (Unauthenticated Visitor)
Guests are visitors who have not registered. They have access only to a pre-populated demo dashboard with static sample data. The purpose of the guest experience is to demonstrate the platform's value and encourage registration.
Attribute	Value
Authentication Required	No
Data Access	Read-only sample data; no personal data
Permissions	View demo dashboard; access registration and login pages
Session Persistence	None (stateless)
Technical Proficiency	Any — general internet users

2.3.2 Registered User (Authenticated End-User)
Registered users are the primary audience. They interact with the full feature set after completing registration and email or phone verification.
Attribute	Value
Authentication Required	Yes (email + password or phone + OTP)
Data Access	Own data only — isolated by userId
Permissions	All end-user features (CRUD transactions, budgets, reports, etc.)
Session Persistence	JWT with 7-day expiry; 30-day refresh token
Technical Proficiency	Basic to intermediate; comfortable with financial concepts

2.3.3 Administrator
Administrators manage the platform itself. They cannot see individual users' personal financial data (transactions, balances) to protect financial privacy, but can manage user accounts, categories, and view aggregate platform analytics.
Attribute	Value
Authentication Required	Yes (elevated role with mandatory 2FA / TOTP)
Data Access	Platform-level: user accounts (non-financial), categories, aggregate stats
Permissions	All registered user permissions + admin panel access
Session Persistence	JWT with 1-hour expiry; mandatory TOTP challenge
Technical Proficiency	Advanced; familiar with platform administration
2.4 Operating Environment
    • Web browsers: Chrome 110+, Firefox 110+, Safari 15+, Edge 110+
    • Minimum screen resolution: 320 px wide (mobile-first responsive design)
    • Internet connection: Required for all features (no offline mode in v1.0)
    • Server OS: Linux (Ubuntu 22.04 LTS recommended)
    • Runtime: Node.js 18 LTS
    • Database: PostgreSQL 15+
    • Cache: Redis 7+
    • Containerization: Docker + Docker Compose
    • Hosting: Cloud platform (AWS, GCP, or Azure) behind NGINX reverse proxy
2.5 Design and Implementation Constraints
    • All user-facing text must support Arabic (RTL) and English (LTR) with dynamic locale switching without page reload.
    • All financial data is stored and transmitted in the user's preferred currency (selected at account level in MVP).
    • The application must use HTTPS exclusively; HTTP connections shall be redirected.
    • Passwords must be hashed using Argon2id (preferred) or BCrypt with a minimum cost factor of 12. Plaintext passwords must never be stored or logged.
    • No payment processing or real banking API integration is in scope for version 1.0.
    • PDF export must be generated server-side to ensure consistent formatting.
    • All API endpoints must follow RESTful conventions and return JSON.
    • The system must comply with GDPR principles: data minimization, right to erasure, and consent logging.
    • The system must use decimal arithmetic exclusively for all monetary calculations (no floating-point).
2.6 Assumptions and Dependencies
    • Users have a reliable internet connection with minimum 1 Mbps bandwidth.
    • Email delivery depends on a third-party SMTP provider (e.g., SendGrid, Mailgun). Outages may delay notifications but shall not affect core functionality.
    • SMS delivery for OTP depends on a third-party SMS gateway (e.g., Twilio). Fallback to email OTP must be implemented.
    • The PDF export engine (WeasyPrint or Puppeteer) must be self-hosted or appropriately licensed.
    • Exchange rate data (if multi-currency display is added in a future version) will require a third-party rates API.
    • All monetary values are stored in a single currency per user account in MVP (Assumption A2).
    • Monthly summaries are generated automatically on a scheduled basis (Assumption A4).

5. Business Rules
ID	Rule
BR-001	Every transaction must belong to exactly one account.
BR-002	Expense transactions must belong to a category. Income transactions may optionally be categorized.
BR-003	Budgets are category-specific and scoped to a single calendar month.
BR-004	Only one active budget is permitted per user per category per month.
BR-005	Fund transfers create two atomically linked transaction records (one debit, one credit). If either fails, both are rolled back.
BR-006	Users may only access their own financial data; cross-user data access is strictly forbidden.
BR-007	Budget alerts trigger at 80% (warning) and 100% (exceeded). The 80% alert is sent at most once per budget per month. The 100% alert is resent every 3 days while the budget remains exceeded.
BR-008	Recurring transactions are executed by a scheduled job at 00:01 server time daily. If a recurring transaction fails, the user is notified and it is marked as 'Failed'.
BR-009	Editing a recurring transaction template applies only to future occurrences; past generated transactions remain unchanged.
BR-010	Deleting a recurring template does not delete previously generated transaction instances.
BR-011	The initial balance of an account is recorded as a special 'Initial Balance' income transaction and cannot be directly edited.
BR-012	Account type cannot be changed after creation.
BR-013	Deleting an account with associated transactions is forbidden; archiving is required instead.
BR-014	Transfer transactions cannot be independently edited; they must be modified via the transfer flow.
BR-015	All monetary values must be stored with exactly 2 decimal places using decimal arithmetic (not floating-point).
BR-016	Dashboard income and expense totals exclude transfer transactions to prevent double-counting.
BR-017	Savings Rate is calculated as: ((Total Income − Total Expenses) / Total Income) × 100. Displayed as 0% if Total Income is zero.
BR-018	Admin role cannot be self-assigned; it must be granted by an existing Administrator.
BR-019	There must always be at least one active Administrator account on the platform.
BR-020	Monthly summary emails are sent on the 1st of each month regardless of individual notification preferences; users may unsubscribe via email link per GDPR/CAN-SPAM requirements.
BR-021	Transaction dates may be in the past or future, but may not precede the associated account's creation date.
BR-022	Budget calculations include all expense transactions of the matching category within the calendar month, regardless of when the budget record was created.

6. External Interface Requirements
6.1 User Interface Standards
    • Mobile-first responsive design using a 12-column grid system.
    • Breakpoints: Mobile (<768 px), Tablet (768–1024 px), Desktop (>1024 px).
    • Supports both LTR (English) and RTL (Arabic) layouts, switching dynamically on language change.
    • Minimum contrast ratio of 4.5:1 for normal text and 3:1 for large text (WCAG 2.1 Level AA).
    • All interactive elements have visible focus indicators for keyboard navigation.
    • Form fields display inline validation errors immediately after losing focus.
    • Loading states indicated by skeleton screens or spinners; never blank content areas.
    • Destructive actions (delete, archive) require a confirmation dialog before execution.
    • Touch-friendly controls with minimum 44×44 px touch targets on mobile.
    • Follows Material Design 3 component guidelines.

6.3 API Design
6.3.1 Conventions
    • Base URL: https://api.pft.example.com/v1
    • All requests and responses use JSON (Content-Type: application/json).
    • Authentication via Bearer token in Authorization header.
    • Standard HTTP status codes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 429 Too Many Requests, 500 Internal Server Error.
    • All list endpoints support ?page=, ?limit=, ?sort=, ?order= query parameters.
    • All error responses follow the structure: { error: { code, message, details } }.

9.2 High-Level Architecture
Version 1.0 shall be implemented as a modular monolith with clear internal service boundaries. This provides simpler deployment, faster development velocity, and a clear migration path to microservices when scale demands it.
Layer	Components
Frontend (React SPA)	Pages, Components, Custom Hooks, Redux Slices, API Service Layer (Axios), i18n resources
NGINX Reverse Proxy	SSL termination · Static asset serving · Rate limiting · /api/* routing
REST API Layer (Express)	Route Modules per resource (auth, accounts, transactions, budgets, reports, notifications, admin)
Business Services Layer	Auth Service · Account Service · Transaction Service · Budget Service · Reporting Service · Notification Service · Recurring Transaction Service
Data Access Layer (ORM)	Repository pattern wrapping database models; handles queries, aggregations, migrations
PostgreSQL Database	Primary relational store for all application data
Redis Cache	JWT blacklist · Rate limiter state · Short-lived session data
Scheduler (node-cron)	recurringTransactionJob (daily 00:01) · monthlyEmailJob (1st of month) · budgetCheckJob (post-expense)
PDF Generation Service	Puppeteer / WeasyPrint sidecar; accepts report payload, returns PDF buffer
Email Service (SendGrid)	Transactional emails: verification, alerts, summaries, password reset
Object Storage (S3)	Receipt image storage; presigned upload URLs; CDN-served download
9.3 Deployment Architecture
    • All components are containerized via Docker.
    • NGINX acts as the entry point: routes /api/* to the Node.js API container and serves the React SPA static files.
    • PostgreSQL hosted on a managed cloud service (e.g., AWS RDS) for reliability, automated backups, and point-in-time recovery.
    • Redis hosted on a managed service (e.g., AWS ElastiCache) or self-managed container.
    • AWS S3 stores receipt images; CloudFront CDN serves images with long-lived cache headers.
    • CI/CD pipeline (GitHub Actions): on push to main — run tests, build Docker image, push to registry, deploy to production.
    • Staging environment is mandatory before any production deployment.
9.4 Evolution Path to Microservices (v2.0+)
The Notification Service, PDF Export Service, and Recurring Transaction Scheduler are already isolated as internal modules and are the recommended candidates for extraction as independent microservices when traffic and scale require it.
